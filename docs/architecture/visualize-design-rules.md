# Visualize Design Rules

This note defines the default craft rules for HTML artifacts generated by `visualize`.

The goal is not just valid HTML. The goal is HTML that looks intentionally designed by default.

## Design Goal

Generated artifacts should feel:
- calm
- readable
- structured
- polished
- modern without being trendy or noisy

The baseline should resemble a strong product/design system with good defaults, not an over-styled AI demo page.

## Core Principles

### 1. Clarity over novelty
Prefer visual clarity and hierarchy over decorative effects.

### 2. One primary reading path
Every artifact should have a clear main flow. Avoid layouts where everything competes equally for attention.

### 3. Summary first, detail second
Dense artifacts should surface summaries, then allow expansion into detail.

### 4. Consistency beats variety
Use a small set of layout and component patterns repeatedly rather than inventing a new card style for every section.

### 5. Beautiful by restraint
Avoid gradients, shadows, borders, colors, and animations unless they serve hierarchy or meaning.

## Layout Rules

- Use an 8px spacing grid
- Prefer a strong max-width for reading-heavy content
- Use whitespace to separate sections instead of excessive border chrome
- Use multi-column layouts only when the information type genuinely benefits
- Keep sticky navigation simple and non-intrusive

## Typography Rules

- Use one strong sans-serif stack by default
- Body text must remain comfortably readable
- Establish a clear type scale for title, section heading, subsection, body, and meta text
- Long prose should optimize line length for reading, not fill the whole screen width
- Muted text should remain readable, not ghosted into invisibility

## Color Rules

- Use a restrained neutral base palette
- Use one accent color consistently
- Use semantic colors only for true meaning (risk, warning, success, neutral)
- Never use many unrelated accent colors unless the visualization mode specifically requires category encoding
- Dark mode must preserve contrast and hierarchy

## Component Rules

Common reusable patterns:
- summary cards
- section cards
- status chips / tags
- callout blocks
- collapsible details
- comparison tables
- timeline / phase blocks

Avoid:
- too many card variants
- nested card-inside-card-inside-card layouts
- gratuitous glassmorphism
- decorative-only icons everywhere

## Interaction Rules

- Progressive disclosure is preferred for dense content
- Collapsible sections are good when they preserve scanning clarity
- Hover effects should be subtle
- Keyboard and responsive behavior should remain sane even for static artifacts
- Interactions should help comprehension, not show off

## Mode-Specific Guidance

### Structure artifacts
- emphasize hierarchy, grouping, and navigation
- use callouts for risks, assumptions, and decisions
- avoid turning long prose into overloaded node labels

### Mockups
- prioritize believable product layout over documentation chrome
- follow product-surface conventions: nav, cards, panels, tables, forms, and states
- avoid fake lorem-ipsum-heavy hero shots unless the user explicitly wants marketing-style output

### Explainers
- use strong narrative sequencing
- summary, comparison, and recommendation sections should be visually distinct
- ideal for sharing with another human who did not author the source document

## Anti-Patterns

Reject outputs that look like:
- a giant wall of undifferentiated cards
- a rainbow dashboard with no semantic meaning
- tiny body text and oversized decorative headings
- excessive shadows, gradients, or rounded corners everywhere
- a mind map forced onto content that is not naturally branch-shaped
- a UI mock that looks like a toy instead of a plausible product screen

## Relationship to Renderer Choice

These rules are orthogonal to mode selection.

Renderer choice answers:
- what visual form best fits this artifact?

Design rules answer:
- how do we make that form consistently high quality?

Both are required.
